# Chat Resume 改进路线图

## 文档目标

把当前项目的主要改进空间收敛成一份可执行 roadmap。

本文不追求“理想架构全景图”，而是优先回答三件事：

1. 现阶段最该修什么
2. 为什么这件事优先级更高
3. 做完以后如何验证它真的变好了

---

## 当前判断

项目当前状态不是“功能缺失严重”，而是：

- 主流程已经可用
- 后端测试基础较完整
- 前端类型检查可通过
- 但安全、规格一致性、状态恢复、数据生命周期和工程质量门开始出现结构性风险

这意味着下一阶段的重点不应该是继续平铺新功能，而应该是先压住系统复杂度。

---

## 成功标准

本轮改进的总体成功标准：

1. 认证和会话机制达到基本生产可用，不再依赖脆弱的前端 token 存储方案
2. AI 简历优化链路只有一套明确语义，文档、接口、前端行为一致
3. Agent 确认和恢复链路形成完整闭环，断流或超时后可以继续
4. 简历删除、导出、布局保存等关键数据流没有明显脏数据和假能力
5. 项目具备稳定的 CI 质量门，而不是只靠本地手工检查

## P0：必须优先处理

### 1. 认证与会话安全加固

#### 问题

- `access_token` 保存在 `localStorage`
- 前端再把 token 写入普通 cookie 供 middleware 使用
- middleware 只判断“是否存在 cookie”，不判断 token 是否有效
- 后端还存在默认 `SECRET_KEY`

这套方案在开发期可接受，但在生产环境风险偏高。

#### 目标

把认证机制升级到“基本生产可用”。

#### 建议改动

1. 改为后端下发 `httpOnly`、`secure`、`sameSite` 明确的 cookie
2. middleware 不再只检查 cookie 是否存在，而是依赖服务端校验结果或改成纯服务端守卫
3. 禁止生产环境使用默认 `SECRET_KEY`
4. 刷新 token 增加 rotation 策略，旧 token 失效
5. 明确登出语义，保证客户端和服务端状态同步失效

#### verify

- 未登录访问受保护页面会被重定向
- 伪造 cookie 无法进入受保护页面或调用受保护接口
- 刷新 token 后旧 refresh token 不能继续使用
- 生产配置缺少 `SECRET_KEY` 时服务启动失败

---

### 2. 统一 AI 简历优化语义

#### 问题

- 文档仍描述 proposal 流程
- 代码里 proposal 表已经移除
- 前端还保留旧 hook，指向已不再是主链路的接口
- 当前用户不容易明确知道“AI 是否已经改动了简历”

#### 目标

整个项目只保留一套简历优化语义。

#### 建议改动

1. 明确选型：
   - 方案 A：直接修改正文，工具级确认后落库
   - 方案 B：统一先 proposal，再 apply
2. 删掉未使用的旧接口、旧 hook、旧文档描述
3. 更新 `product-spec`，让文档只描述当前真实链路
4. 给前端补充明确文案：
   - “已修改并保存”
   - “仅生成建议，未保存”

#### verify

- 前端只有一个 AI 优化入口
- 仓库里不再存在未使用的旧优化 hook 或废弃 API 引用
- 产品文档中的流程图和实际接口一致
- 用户可以清楚区分“建议”与“已落库”

---

### 3. 打通 Agent 确认与恢复闭环

#### 问题

- 后端已有 session store 和恢复接口
- 前端当前没有真正消费恢复能力
- `tool_pending` 超时后主要是本地 UI 状态变化，不是完整恢复流

这会导致“后端可恢复，前端不可恢复”的半成品状态。

#### 目标

让 resume agent 的确认链路具备真实可恢复性。

#### 建议改动

1. 前端持久化当前 `session_id`
2. 页面刷新、断流、重新进入编辑页时检查是否存在未完成 session
3. 对 `waiting_confirmation` / `paused` 状态提供恢复入口
4. 把恢复后的事件回放到 UI，而不是只返回最终结果
5. 为超时和断流场景定义明确 UX

#### verify

- 在 `tool_pending` 时刷新页面，可以恢复到待确认状态
- 流式连接中断后，用户可以继续确认或取消
- 后端 session 状态和前端展示状态一致
- 新增 e2e 或集成测试覆盖“确认后恢复执行”场景

---

### 4. 修正导出能力的真实性

#### 问题

- 前端主流程只显式使用 PDF 导出
- 后端 `export_to_docx()` 代码看起来并未真正生成 Word 文档
- 文档里仍声称支持 `PDF`、`DOCX`、`HTML`

这是典型的“对外承诺大于真实能力”。

#### 目标

导出功能只暴露真实可用的能力。

#### 建议改动

1. 校验当前 `docx` 导出结果是否为真实可打开的 Word 文件
2. 若不可用：
   - 立即修为真正的 `python-docx` 实现，或
   - 暂时从前端和文档中移除 `docx`
3. 给导出功能增加后端测试
4. 给前端导出按钮和返回文案补足失败提示

#### verify

- 导出的 `pdf`、`docx`、`html` 均能被对应程序正常打开
- 导出接口至少有一个成功用例测试
- 前端不会展示不可用格式

## P1：下一阶段应完成

### 5. 修复数据生命周期与级联删除

#### 问题

- 删除简历时只显式处理了部分关联数据
- `Resume` 与 `InterviewSession`、`ResumeChatMessage` 的关系约束不完整
- 未来切到更严格的 Postgres/FK 约束后，容易出现删除失败或脏数据

#### 目标

所有核心实体删除时，关联数据行为明确且一致。

#### 建议改动

1. 明确 `Resume` 删除时哪些数据应级联删除
2. 补 ORM relationship 和 cascade 配置，或在 service 层显式清理
3. 补外键约束一致性检查
4. 为删除链路加测试：
   - 删除简历
   - 删除后聊天记录消失
   - 删除后面试记录消失

#### verify

- 删除简历后无孤儿记录
- SQLite 和 Postgres 下删除行为一致
- 集成测试覆盖关联清理

---

### 6. 建立真正的 CI 质量门

#### 问题

- 后端测试存在，但 GitHub workflow 里没有标准 CI 门禁
- 前端 `lint` 尚未完成配置
- 现在的质量控制更偏手工执行

#### 目标

每次提交都能自动验证基础质量。

#### 建议改动

1. 新增标准 CI workflow
2. 至少执行：
   - backend pytest
   - frontend type-check
   - frontend lint
3. 完成前端 ESLint 配置，避免 `next lint` 进入交互式初始化
4. 视耗时情况增加最小 smoke e2e

#### verify

- 新 PR 会自动跑 CI
- CI 失败会阻止低质量改动继续合并
- `npm run lint` 在本地和 CI 都可直接运行

---

### 7. 清理旧代码与无效依赖

#### 问题

- 旧优化 hook 仍存在
- 文档仍有已失效的数据模型描述
- `uv` 是主依赖管理工具，但仓库里仍保留 `requirements.txt`
- `redis` 等依赖已声明，但未形成清晰主流程职责

#### 目标

减少“看起来有、实际上不用”的系统噪音。

#### 建议改动

1. 删除未使用的 hook、接口引用、文档残留
2. 明确 `requirements.txt` 是否还需要维护
3. 清理未进入主链路的依赖，或补上实际用途说明
4. 把依赖来源统一到 `uv`

#### verify

- 删除无用文件后构建、测试、类型检查仍通过
- 新同学进入项目时不会被旧链路误导
- 依赖清单与实际使用一致

## P2：结构优化与体验增强

### 8. 拆分超大文件，降低变更耦合

#### 问题

- 编辑页单文件超过 1500 行
- 面试 endpoint 和 AI endpoint 体量都偏大
- 前端 API 客户端也已经开始积累太多职责

#### 目标

让关键模块的职责边界更清晰，降低回归风险。

#### 建议改动

1. 将编辑页拆为：
   - 页面容器
   - 编辑器区域
   - 预览区域
   - Agent 面板
   - 面试面板
2. 将面试 endpoint 拆为：
   - session lifecycle
   - answer/evaluation
   - report
3. 将前端 API 按 domain 分模块导出
4. 补跨模块测试，确保拆分不是纯搬文件

#### verify

- 核心大文件明显缩小
- 单个模块修改的影响范围更小
- 拆分后现有测试仍通过

---

### 9. 提升可观测性和线上排障效率

#### 问题

- 当前已有 request id、Langfuse、Sentry 基础
- 但关键业务事件的观测面仍不够完整
- 前端日志较多，缺少生产环境分级策略

#### 目标

让“问题发生后能定位”成为系统默认能力。

#### 建议改动

1. 为核心链路统一埋点：
   - 登录
   - 上传解析
   - Agent run
   - 工具确认
   - 导出
   - 面试 start/answer/end
2. 区分开发日志和生产日志
3. 给关键失败路径定义 error code
4. 为长耗时操作增加结构化 timing

#### verify

- 任一失败请求都能关联 request id / user id / session id
- 常见线上问题可通过日志快速定位
- 生产环境无大量噪音 `console.log`

---

### 10. 体验层补强

#### 问题

- 一些流程技术上能完成，但用户感知不够明确
- 尤其是自动保存、AI 修改确认、面试恢复、导出失败等状态

#### 目标

把“系统内部状态”翻译成“用户可理解状态”。

#### 建议改动

1. 自动保存状态统一文案和视觉反馈
2. AI 工具确认增加更明确的结果提示
3. 面试 session 补充恢复提示和最近进度提示
4. 导出失败时给出可操作建议，而不是泛化报错
5. 设置页补充账号状态、会话状态、环境诊断入口

#### verify

- 用户在关键流程中知道当前发生了什么
- 失败提示可指导下一步操作
- 相关页面的 e2e 用例覆盖核心状态切换

## 推荐执行顺序

### 第一阶段

1. 认证安全加固
2. 统一 AI 优化语义
3. 打通 Agent 恢复闭环
4. 修正导出真实性

### 第二阶段

1. 修复删除链路和数据生命周期
2. 建立标准 CI
3. 清理旧代码和依赖

### 第三阶段

1. 拆大文件
2. 强化可观测性
3. 打磨体验层

---

## 建议的里程碑定义

### Milestone 1：Production Baseline

完成后应达到：

- 登录和会话方案可用于正式环境
- AI 主链路语义清晰
- Agent 确认流程可恢复
- 导出能力真实可用

### Milestone 2：Stable Delivery

完成后应达到：

- 数据生命周期一致
- CI 自动化运行
- 旧链路基本清除

### Milestone 3：Scalable Iteration

完成后应达到：

- 大模块已拆分
- 线上排障效率明显提升
- 新功能迭代不再频繁碰撞旧债
