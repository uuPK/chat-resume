# 测试工程视角：这个项目值不值得改

更新时间：2026-05-13

## 一句话结论

值得改，但不要大改。

这个项目已经有不少测试，说明不是烂摊子。真正的问题是：测试没有变成稳定的发布保护网，关键用户链路也还没有被完整守住。

## 我看到的事实

我在当前项目里实际跑了这些检查：

```bash
cd backend && uv run --extra dev python -m pytest tests -q
```

结果：`320 passed`

```bash
cd backend && uv run basedpyright
```

结果：`0 errors`

```bash
cd frontend && npm run type-check
cd frontend && npm run build
```

结果：都通过。

另外，后端测试覆盖率是 `66%`。

这说明项目基础还可以，不是“不能改”，而是“应该更有顺序地改”。

## 最值得改的 3 件事

### 1. 先加 CI，防止回归

现在 `.github/workflows/` 里主要是 Claude 相关 workflow，没有看到强制跑测试的 CI。

这很危险：本地测试能过，不代表 PR 合进去之前一定有人跑。

建议先加一个最小 CI，只做四件事：

```bash
cd backend && uv run --extra dev python -m pytest tests -q
cd backend && uv run basedpyright
cd frontend && npm run type-check
cd frontend && npm run build
```

这是当前最划算的改进。

### 2. 补一条核心用户流程测试

这个产品最重要的流程是：

```text
登录 -> 上传/新建简历 -> 输入 JD -> Agent 给修改建议 -> 用户确认 -> 简历被更新 -> 导出
```

现在有一些前端 e2e，但还不够像“用户真的完整走了一遍”。

建议只补一条最小流程，不要一次性测所有页面。

### 3. 重点保护 Agent、导出、语音这几块

覆盖率里有些高风险模块偏低：

- LLM/Agent 调用
- 简历导出
- 简历解析
- 语音面试
- 数字人/实时语音服务

这些地方一坏，用户马上能感知。

所以后续补测试时，不要平均用力。优先补这些会直接影响体验的地方。

## 不建议做什么

不建议现在追求很高的覆盖率，比如 90%。

这个项目有模型、支付、OAuth、语音、数字人、导出文件等外部依赖。为了覆盖率强行写很多 mock，后面会很难维护。

也不建议把真实模型 eval 放进每个 PR 的强制检查。模型输出会波动，适合手动跑或定时跑，不适合阻塞每次提交。

## 最小行动计划

第一步：加 CI。

第二步：补一条核心用户流程 e2e。

第三步：给 Agent、导出、语音这些高风险模块补针对性测试。

只做这三步，项目的回归风险就会明显下降。

## 最终判断

从测试工程师角度看，这个项目值得继续改进。

但正确方式不是“大规模测试改造”，而是先把已有测试跑起来、自动化起来，再围绕最重要的用户链路补测试。
