# 当前最有价值做的五件事

本文档从产品经理视角给出 chat-resume 当前最值得推进的五个方向。判断依据是：用户能否更快获得求职结果、是否能形成可验证的价值闭环、是否能提高留存和付费转化，以及是否适合用最小功能切片快速验证。

## 当前产品判断

chat-resume 已经有一条完整的基础链路：上传简历、补充 JD、Agent 定向优化、确认修改、导出简历、创建模拟面试、保存面试记录。现在最重要的不是继续横向堆功能，而是把这条链路变成用户能看懂、能信任、愿意反复使用、愿意付费的闭环。

现阶段的核心机会是把“AI 帮你改简历”升级成“围绕一个目标岗位持续提高命中率和面试表现”。

## 1. 做新用户首条成功路径

优先级：P0

目标：让新用户第一次进来时，在 5 分钟内完成一次完整的求职准备动作：上传简历、填写或上传 JD、看到岗位差距、接受至少一条修改、导出或进入面试。

为什么最有价值：当前产品能力已经存在，但首次路径仍然依赖用户自己理解“下一步该做什么”。如果用户第一次体验没有立刻看到岗位差距和修改收益，后面的 Agent、面试、付费能力都很难被感知。

最小功能切片：

- 在简历中心空状态增加“上传简历并匹配岗位”的单一路径。
- 上传完成后直接引导用户补充目标岗位和 JD，而不是让用户停在编辑器里摸索。
- Agent 第一次只做一件事：输出 3 个最大岗位差距，并建议 1 条可确认修改。
- 用户确认修改后展示“已完成第一轮岗位优化”的结果状态。

验收信号：

- 新用户从注册到首次 Agent 修改确认的完成率。
- 首次上传后 10 分钟内进入编辑页并发送 Agent 消息的比例。
- 首次优化后导出或进入面试的比例。

暂不做：

- 不做复杂的新手任务系统。
- 不做多步骤教学弹窗。
- 不在首轮塞入所有功能介绍。

## 2. 做 JD 匹配证据链

优先级：P0

目标：让用户清楚知道“为什么要这样改”，并能看到优化前后的岗位匹配变化。

为什么最有价值：简历优化产品的信任来自证据，而不是 AI 的自然语言建议。当前编辑页已有 diff 确认能力，但还缺少围绕 JD 的整体解释：哪些关键词命中、哪些要求缺失、哪些经历被强化、哪些内容存在事实风险。

最小功能切片：

- 在 Agent 完成一轮优化后生成一个“岗位匹配摘要”。
- 摘要只包含四类信息：已命中关键词、缺失关键词、优化前后变化、需要用户补充事实的点。
- 每条修改都关联一个 JD 证据或简历证据。
- 对无法确认的量化结果标记为“需要补充证据”，避免 AI 编造。

验收信号：

- 用户接受 Agent 修改的比例。
- 用户拒绝修改时的主要原因是否从“不可信”变为“我不想这样表达”。
- 用户在导出前补充目标岗位、JD 或量化结果的比例。

暂不做：

- 不先做复杂 ATS 总分。
- 不做无法解释的黑盒评分。
- 不让 AI 自动补充未经用户确认的数字和经历。

## 3. 把简历中心变成求职工作台

优先级：P1

目标：让用户一眼知道每份简历正在服务哪个岗位、优化到什么阶段、下一步该做什么。

为什么有价值：当前简历中心主要是文件列表和预览。对真实求职场景来说，用户管理的不是“简历文件”，而是“岗位机会”：某家公司、某个职位、一份 JD、一轮优化、一场面试、一次导出。

最小功能切片：

- 每张简历卡片展示目标公司、目标岗位、最近优化时间、当前状态。
- 状态只保留 4 个：未补 JD、待优化、已优化、待面试。
- 卡片主按钮根据状态变化：补 JD、开始优化、导出简历、开始面试。
- 简历中心顶部增加一个“下一步优先处理”的区域。

验收信号：

- 老用户回到简历中心后的下一步点击率。
- 简历创建后 24 小时内完成第二次动作的比例。
- 多份简历用户的回访率和导出率。

暂不做：

- 不做完整 CRM。
- 不做投递进度管理。
- 不做复杂看板拖拽。

## 4. 打通面试反馈到简历优化

优先级：P1

目标：让模拟面试不只是一次练习，而是反向发现简历问题，并沉淀成下一轮简历优化建议。

为什么有价值：简历和面试天然是一个闭环。用户在面试中回答不出来的问题，往往对应简历中的空泛表达、缺少量化结果、项目真实性不足或岗位匹配弱点。把面试记录回流到简历优化，会让产品从一次性工具变成持续准备系统。

最小功能切片：

- 面试结束后生成 3 条“可用于优化简历的反馈”。
- 每条反馈绑定到简历模块：工作经历、项目、技能或求职目标。
- 用户可以一键把反馈发送给 Resume Agent，由 Agent 给出可确认 diff。
- 简历编辑页显示这条建议来自哪场面试和哪个回答。

验收信号：

- 面试结束后回到简历编辑页的比例。
- 面试反馈转化为 Agent 修改确认的比例。
- 同一岗位下完成“优化 -> 面试 -> 再优化”的用户比例。

暂不做：

- 不做大而全的面试报告系统。
- 不做复杂能力画像。
- 不把所有面试文本自动写回简历。

## 5. 明确付费价值边界

优先级：P1

目标：把 Free、Plus、Pro 的差异从“页面上的套餐”变成用户在关键时刻能理解的价值边界。

为什么有价值：项目里已经有定价页、设置页和 PayPal 订阅路径，但商业化价值还需要和核心场景绑定。用户不是为了“订阅”付费，而是为了更多岗位优化、更高质量的 JD 证据链、模拟面试和导出能力付费。

最小功能切片：

- 定义 Free 的可用额度：简历数量、Agent 优化次数、面试次数、导出次数。
- 在用户触达额度时展示 Plus 能解锁的具体结果，而不是泛泛升级提示。
- 设置页展示当前套餐、已用额度、下一次续费或取消入口。
- Plus 只绑定当前最强价值：更多岗位优化、面试反馈回流、PDF 导出。

验收信号：

- 额度触达后的升级点击率。
- 定价页到 PayPal checkout 的转化率。
- 付费用户的导出次数、面试次数和二次优化次数。

暂不做：

- 不急着做企业版。
- 不先做复杂优惠券。
- 不把基础编辑能力放到付费墙后面。

## 推荐执行顺序

1. 新用户首条成功路径。
2. JD 匹配证据链。
3. 求职工作台化的简历中心。
4. 面试反馈回流到简历优化。
5. 付费价值边界和额度触达。

这个顺序的核心逻辑是：先提高首次价值感知，再提高信任，再提高回访，再形成闭环，最后把付费点放在用户已经感受到价值的位置。

## 最近一个可执行切片

建议先做“首次岗位优化切片”：

1. 用户上传简历后，进入目标岗位填写区。
2. 用户填写 JD 后，系统生成 3 条岗位差距。
3. 用户选择其中 1 条，让 Agent 生成一条可确认修改。
4. 用户确认修改后，页面展示“第一轮岗位优化完成”。

这个切片横跨上传、JD、Agent、diff 确认和结果反馈，是最小但完整的产品价值闭环。它能最快验证用户是否真的愿意沿着“目标岗位 -> 简历优化 -> 结果确认”的主路径走下去。
