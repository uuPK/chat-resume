# 国际化架构说明

ChatSume 前端使用 `next-intl` 管理中文和英文文案。当前采用单站点、多语言路由方案，不复制页面，也不维护国内版和海外版两套代码。

## 当前架构

- 语言路由：`/zh` 和 `/en`，配置在 `frontend/src/i18n/routing.ts`。
- 默认语言：中国用户优先中文，其他用户优先英文。
- 自动识别：`frontend/src/proxy.ts` 根据 `NEXT_LOCALE` cookie 和 `Accept-Language` 重定向到对应语言路径。
- 手动切换：`frontend/src/components/i18n/LocaleSwitcher.tsx` 切换语言，并写入 `NEXT_LOCALE` cookie。
- 翻译加载：`frontend/src/i18n/request.ts` 加载所有 namespace，并交给 `NextIntlClientProvider`。
- 页面结构：业务页面只放在 `frontend/src/app/[locale]/` 下，同一套页面服务所有语言。
- 打印页：`/resume/print` 保持非语言路由，避免影响后端 PDF 导出。

## 翻译文件

翻译文件统一放在 `frontend/locales/`：

```text
frontend/locales/
  zh/
    common.json
    auth.json
    dashboard.json
    interview.json
    resume.json
  en/
    common.json
    auth.json
    dashboard.json
    interview.json
    resume.json
```

namespace 的职责：

- `common`：导航、通用按钮、通用状态、通用错误。
- `auth`：登录、注册、OAuth、认证错误。
- `dashboard`：仪表盘、套餐、设置。
- `interview`：面试中心、创建面试、语音面试。
- `resume`：简历中心、编辑器、预览、布局控制。

## 如何新增语言

1. 在 `frontend/src/i18n/routing.ts` 的 `locales` 中加入语言代码，例如 `ja`。
2. 新建 `frontend/locales/ja/`，并补齐当前所有 namespace 文件。
3. 确认每个 JSON 文件的 key 与 `zh`、`en` 完全一致。
4. 如果面试接口需要语言代码，在 `toInterviewLanguage` 中增加映射。
5. 运行 `npm run type-check`、`npm run build` 和 i18n e2e 测试。

## 如何新增 translation key

1. 先判断文案属于哪个 namespace，不要把所有 key 放进 `common`。
2. 同时修改 `zh` 和 `en` 两份 JSON。
3. key 使用稳定的业务含义，避免使用页面上的完整句子作为 key。
4. 组件里使用 `useTranslations(namespace)`，服务端组件使用 `getTranslations`。
5. 新增动态变量时使用命名占位符，例如 `{pages}`、`{title}`。

## 维护规则

- 组件里不要硬编码面向用户的中英文文案。
- 不要为不同语言复制页面、组件或业务逻辑。
- 新增页面时默认放在 `frontend/src/app/[locale]/`。
- 新增导航链接时使用 `frontend/src/i18n/navigation.ts` 导出的 `Link` 和 `useRouter`。
- 认证、接口、状态管理不应依赖显示语言，只在确实需要时传递 locale。
- 后端兼容路径不要随意加语言前缀，例如当前 `/resume/print`。
- 提交前运行中文硬编码扫描：

```bash
rg -n "'[一-龥][^']*'|\"[一-龥][^\"]*\"|>[[:space:]]*[一-龥]" frontend/src -g '*.tsx' -g '*.ts'
```

## 最佳实践

- 文案按业务域拆分，避免翻译文件过大。
- 修改 UI 时同步检查移动端，尤其是英文更长的按钮和导航。
- 错误提示需要保持可操作，不要只翻译成泛化失败。
- toast、placeholder、aria-label、空状态、loading 状态都属于必须国际化的文案。
- URL 保持显式语言前缀，利于 SEO、分享和用户理解当前语言环境。
