## 问题陈述

当前简历 Agent 的评估系统已经能跑离线用例、计算关键词提升、工具调用 F1、LLM-as-Judge 和面试链路完成率，但它更像一次性指标脚本，还没有形成稳定的“评估 -> 定位失败 -> 修改 Agent -> 回归验证”闭环。

从使用者视角看，主要痛点是：

- 评测结果能说明平均表现，但很难快速知道具体失败属于 prompt、工具选择、内容质量、事实安全、输入不足处理还是运行时稳定性问题。
- 用例定义里有 `must_contain_keywords`、`forbidden_content`、`expect_refusal` 等字段，但当前评分器没有系统性消费这些期望，导致 case 里的一部分评估意图没有真正进入分数。
- 当前输出缺少门禁阈值、失败分组、趋势对比和可直接指导下一轮修改的建议。
- LLM-as-Judge 有成本和稳定性问题，缺少默认可重复的本地客观评估路径。
- 评估体系目前覆盖 Resume Agent 和面试链路，但二者的结果没有统一到一套证据和回归视图里。

本 PRD 按“简历 Agent 评估系统”理解用户提到的 `AGTNDT`。如果 `AGTNDT` 是项目里的专有缩写，后续实现时仅替换展示命名，不影响模块边界。

## 解决方案

把现有 `eval` 能力升级为一个面向迭代的评估系统：它不仅产出分数，还产出可行动的失败诊断、门禁结论、趋势摘要和证据包输入。

第一阶段采用曳光弹切片：复用已有 `eval_results.json` 和 `test_cases.json`，新增一条无需真实 LLM 调用的本地分析链路，输出：

- 每条 case 的客观断言结果。
- 按失败类型聚合的诊断。
- 是否通过最低质量门禁。
- 可读 Markdown 摘要和结构化 JSON 摘要。

后续再扩展到完整离线运行、LLM Judge 标定、历史趋势、CI 门禁和面试链路统一报告。

## 用户故事

1. 作为正在迭代 Resume Agent 的开发者，我希望能基于已保存的评测结果快速生成确定性评估摘要，从而不用消耗模型成本也能发现回归。
2. 作为开发者，我希望每个失败用例都展示失败的期望、实际工具调用、关键词命中、禁用内容命中和决策规则状态，从而判断应该修改 prompt、工具、样本还是评分逻辑。
3. 作为开发者，我希望评分器系统性消费每个用例里显式声明的期望，从而让 `must_contain_keywords`、`forbidden_content`、拒绝期望和决策期望真正参与评分，而不是只作为注释存在。
4. 作为开发者，我希望失败能按稳定分类聚合，从而看出系统主要失败在工具选择、内容事实约束、过度追问、安全边界还是 JD 匹配。
5. 作为产品负责人，我希望看到单一的质量门禁摘要，从而判断当前 Agent 是否达到发布、演示或继续迭代的最低标准。
6. 作为产品负责人，我希望评估报告能接入现有证据包，从而让 README、简历项目表述和面试话术使用同一套指标口径。
7. 作为开发者，我希望门禁阈值配置在评分逻辑之外，从而让 CI、本地调试和完整基准评测使用不同通过标准。
8. 作为开发者，我希望每次评测都能记录时间、模型和配置元数据，从而把本次结果与历史基线对比。
9. 作为开发者，我希望默认运行确定性检查，并把 LLM Judge 设为可选能力，从而保持日常反馈链路快速且低成本。
10. 作为维护者，我希望新增评测用例时先做 schema 校验，从而避免格式错误或缺失期望悄悄削弱基准集。
11. 作为维护者，我希望 Resume Agent 指标和面试链路指标使用同一套摘要契约，从而在一个视图里看到端到端产品质量。
12. 作为采用 Agentic 开发方式的开发者，我希望先完成一个很小的端到端评估切片，再扩展大规模重构，从而尽早验证方向并降低返工。

## 实现决策

- 构建一个小型评估分析模块，读取已有运行结果和用例定义，并产出标准化的逐用例评估结果。第一阶段不调用 Agent，也不调用任何 LLM。
- 保持当前运行器继续负责执行 Resume Agent 用例。不要把执行、评分和报告生成塞进一个大型脚本。
- 将评分逻辑拆成多个指标评估器，并保持稳定输入输出结构：用例元数据、运行结果、指标结果、失败原因和可选证据。
- 保留现有指标：
  - 执行成功率
  - JD 关键词覆盖率增量
  - 工具调用正确性 F1
  - optimize-first 决策规则
  - 显式启用时的 LLM-as-Judge
  - 延迟和 fallback 触发率
- 增加确定性用例期望检查：
  - 最终简历是否包含必需关键词
  - 最终简历和 Agent 回复是否避开禁用内容
  - 拒绝或温和拒绝行为是否符合预期
  - 期望只分析不修改的用例是否没有意外工具调用
  - 输入缺失或高风险输入时的追问是否具体且简洁
- 增加稳定的失败分类：
  - `execution_error`
  - `tool_mismatch`
  - `missing_required_keyword`
  - `forbidden_content`
  - `decision_rule_failure`
  - `unsafe_fabrication_risk`
  - `instruction_miss`
  - `quality_judge_low`
  - `latency_or_fallback`
- 将门禁判断作为指标结果之上的单独层。示例门禁：
  - 所有用例执行成功
  - 平均工具 F1 高于阈值
  - 禁用内容失败数为 0
  - 所有 optimize-first 用例通过
  - fallback 触发率低于阈值
- 增加输出契约：
  - 面向自动化和趋势追踪的结构化 JSON
  - 面向人阅读的 Markdown 摘要
  - 面向本地运行的精简终端摘要
- 等第一阶段切片稳定后，更新证据包汇总脚本，使其消费改进后的结构化输出。
- 当前继续使用 JSON 维护样本文件。改变存储格式之前，先补 schema 校验。
- 第一轮不引入 Web dashboard。第一轮可用界面是 CLI 输出和生成文件。
- 使用 `uv` 执行后端脚本和管理依赖。

## 测试决策

- 单元测试应使用小型内存对象验证每个确定性指标评估器。
- 单元测试应验证用例期望字段被正确消费：
  - 必需关键词命中和未命中
  - 禁用内容命中和未命中
  - 期望无工具调用的用例在出现意外工具调用时失败
  - 决策规则检查保持现有行为
- 单元测试应验证失败分类映射和门禁聚合。
- 快照类测试应验证 Markdown 报告包含预期章节，JSON 输出包含稳定键。
- 现有 Resume Agent 工具和 harness 测试仍然相关，但不足以覆盖本功能，因为本 PRD 面向评估行为，而不是 Agent 执行行为。
- 现有 eval 脚本应继续可运行：
  - `uv run python ../eval/run_eval.py ...`
  - `uv run python ../eval/score.py ...`
  - `uv run python ../eval/build_evidence_report.py ...`
- 第一阶段切片应能在没有 `OPENROUTER_API_KEY` 的情况下运行，通过分析已提交或本地保存的评测结果完成验证。

## 非目标

- 不在本次评估系统改造中重写 Resume Agent prompt 或工具策略。
- 不构建完整 dashboard UI。
- 不用复杂评审平台替换 LLM-as-Judge。
- 不新增生产分析、用户转化追踪或留存埋点。
- 不修改简历编辑 schema 或面试 API。
- 不创建分支。

## 补充说明

- 建议第一里程碑：实现“确定性评估分析器”，读取当前 39 条用例的已保存结果，生成失败分类和门禁摘要。
- 建议第二里程碑：为评测用例增加 schema 校验，并引入阈值配置。
- 建议第三里程碑：支持两份评估摘要之间的趋势对比。
- 建议第四里程碑：把改进后的摘要接入现有指标证据包。
- 风险：关键词匹配可能鼓励表层关键词堆砌。应把它视为一个信号，而不是最终质量指标。
- 风险：LLM Judge 分数会随模型和供应商漂移。每份摘要都应保存模型名、prompt 版本、运行时间，以及是否启用 judge。
- 风险：一次性增加过多指标会拖慢迭代。第一阶段保持小而确定，拿到反馈后再扩展。
